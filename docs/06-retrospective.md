# 프로젝트 회고

## 1. 가장 막혔던 지점

로컬에서 구동했던 걸 컨테이너에서 구동할 때 수정하는것, 컨테이너에서 구동했던 걸 AWS에 올릴때 수정하는 것.
인스턴스에서 compose up 할때 빌드jar파일이 없다고 나와서 인스턴스에 깃을 클론하고 인스턴스 안에서 프로젝트를 빌드해 jar을 만들려고 시도했는데 빌드를 진행하니 인스턴스 메모리가 부족해서 빌드를 하다말고 인스턴스가 다운되버리는 현상 발생. 인스턴스가 한번 다운되어 버리니 해당 인스턴스 ip로는 ssh로그인 자체가 안되는 문제가 이어서 발생.
문제점을 파악하지 못하고 멘토님의 조언에 따라 인스턴스를 삭제하고 새로 만들자 처음엔 ssh로그인이 잘 되지만, 이후에 jar 빌드를 또 시도하니 인스턴스가 다시 다운됨.
빌드를 진행할 때 ./gradlew clean build -x test --no-daemon를 사용해서 테스트를 안하고, 그래들 데몬을 사용하지 않음으로써 메모리를 절약하여 빌드 성공.
이후에 무사히 진행할 수 있었다.

## 2. 이해가 부족하다고 느낀 부분

EC2를 사용하는 부분을 배울때 예비군을 다녀와 수업을 4일이나 빠지고 말았는데, 이것때문에 독학만 해서인지 프로젝트 내내 EC2를 사용하는데 어려움을 겪었다.
EC2를 사용하는 배포를 따로 더 진행하면서 이해도를 높여야겠다고 생각했다.
구동 환경은 인텔리제이와 vscode를 사용하는 로컬 환경, 컨테이너를 사용하는 로컬 환경, ec2와 aws를 사용하는 배포 환경 3개로 나뉘는데 3개마다 설정파일의 설정값이 달랐다.
이 부분을 구동환경에 맞게 자동으로 선택되도록 했어야했는데 이해도가 부족해 그때그때 하드코딩하는식으로 프로젝트를 진행했다.

## 3. 팀 프로젝트 전에 보완하고 싶은 기술

EC2 배포에 관한 기술과 구동환경에 따라 설정파일을 자동으로 선택하는 기술을 보완해야겠다고 생각했다.

## 4. 혼자 진행하며 느낀 점

강사님이 하는걸 따라할때는 금방금방 됬는데 혼자 하려니 온갖부분에서 어려움이 생겼다. 굉장히 사소한거 하나하나 놓치기 쉬웠다.
프로젝트의 크기를 작게 하고 프로젝트 작성부터 배포까지 일련의 과정을 전부 진행해서 다행이라고 생각했다.
프로젝트를 욕심부려서 크게 잡았으면 EC2와 AWS 쪽에서 크게 막혀서 프로젝트를 마무리짓지 못했을거 같다.
